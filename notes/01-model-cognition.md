# 01 模型认知：工程师视角理解 LLM

更新时间：2026-06-05

## 本课目标

完成本课后，你应该能回答 4 个问题：

1. LLM 和传统确定性程序的根本差异是什么？
2. 为什么同一个 prompt 可能得到不同结果？
3. token、上下文窗口、结构化输出、工具调用分别影响什么工程决策？
4. 一个业务需求应该用 Chat、RAG、Agent、Workflow 还是 Fine-tuning？

## 1. LLM 不是函数，是概率型语言决策器

传统函数的抽象是：

```text
input -> deterministic logic -> output
```

LLM 更接近：

```text
context + instruction + model parameters -> probability distribution -> sampled output
```

工程含义：

- 不要默认模型输出稳定。
- 不要把关键业务状态只放在自然语言里。
- 不要让模型直接决定高风险动作。
- 对关键输出使用 schema、校验、重试、人工确认。
- 用评估集衡量效果，而不是只看几次 demo。

## 2. Token 是 AI 系统的基本成本单位

token 可以粗略理解为模型处理文本的最小片段。中文、英文、代码、标点都会被切成 token。

token 影响：

- 成本：输入和输出 token 通常都会计费。
- 延迟：上下文越长，响应越慢。
- 质量：无关上下文越多，模型越容易分心。
- 架构：长文档不能无脑塞进 prompt，通常需要 RAG、摘要、分块或检索。

工程策略：

- 输入只放与当前任务直接相关的信息。
- 长文档先结构化摘要，再进入模型。
- 代码任务先用 `rg` 定位，再读相关文件。
- 把稳定项目约定沉淀为短文档，避免每次重新解释。

## 3. 上下文窗口不是记忆

上下文窗口是单次请求可见的信息范围，不等于长期记忆。

常见误区：

- “我前面说过”：如果不在当前上下文里，模型未必可见。
- “把所有资料都塞进去”：上下文越多不一定越好。
- “长上下文可以替代 RAG”：长上下文适合少量长材料阅读，不等于可维护知识系统。

工程策略：

- 短期任务状态放在请求上下文。
- 长期知识放在数据库、对象存储、向量库、搜索引擎。
- 用户偏好、项目约定、历史结论需要显式持久化。
- 对回答要求引用来源，降低幻觉风险。

## 4. Temperature 控制随机性，不控制正确性

temperature 越低，输出越稳定；temperature 越高，输出越发散。

建议：

- 分类、抽取、结构化输出：低 temperature。
- 代码修改、技术分析：低到中等 temperature。
- 创意写作、方案发散：中到高 temperature。

注意：

- 低 temperature 不保证正确。
- 高 temperature 不代表聪明。
- 对事实正确性，主要依赖检索、工具、校验和评估，而不是调参数。

## 5. 结构化输出是产品化的关键能力

自然语言适合给人看，结构化输出适合给系统消费。

业务系统更应该要求模型输出：

```json
{
  "intent": "create_task",
  "confidence": 0.82,
  "entities": {
    "title": "补充登录失败日志",
    "priority": "medium"
  },
  "needs_human_confirmation": true
}
```

工程策略：

- 使用 JSON Schema 或强类型结构约束输出。
- 对模型输出做 parse、validate、fallback。
- 不要直接信任模型返回的 ID、金额、权限、SQL、命令。
- 失败时让模型修复结构，而不是让业务代码猜。

## 6. Tool Calling 让模型从“回答”变成“行动”

模型本身不能真的查数据库、发消息、改文件、调用 GitHub。它只能决定“应该调用哪个工具以及传什么参数”，真正执行的是你的系统。

典型工具：

- 查询数据库。
- 搜索知识库。
- 调用业务 API。
- 创建工单。
- 发送消息。
- 读取文件。
- 执行测试。

工程原则：

- 工具参数必须有 schema。
- 工具执行必须有权限控制。
- 工具结果要进入后续上下文。
- 高风险工具需要人工确认。
- 所有工具调用要记录日志。

## 7. Chat、RAG、Agent、Workflow、Fine-tuning 的边界

| 类型 | 解决的问题 | 适合场景 | 不适合场景 |
| --- | --- | --- | --- |
| Chat | 直接基于上下文生成回答 | 问答、解释、改写、总结 | 需要可靠外部事实或执行动作 |
| RAG | 从知识库检索后回答 | 企业知识库、文档问答、客服 | 需要多步骤行动或复杂决策 |
| Agent | 模型选择工具并推进任务 | 自动化助手、研究、代码任务、跨系统操作 | 强确定流程、高风险动作全自动 |
| Workflow | 固定流程中嵌入模型节点 | 审批、内容流水线、运营流程 | 完全开放式探索任务 |
| Fine-tuning | 固定任务的风格、格式、分类能力 | 稳定格式输出、领域表达风格 | 注入大量动态知识 |

判断规则：

- 如果问题只需要生成文本：先用 Chat。
- 如果问题依赖私有知识：用 RAG。
- 如果问题要跨步骤调用工具：用 Agent。
- 如果业务流程本身很明确：用 Workflow。
- 如果同类任务量大且 prompt 很难稳定：再考虑 Fine-tuning。

## 8. 第一个工程判断框架

分析一个 AI 需求时，先填这 10 个问题：

1. 用户输入是什么？
2. 期望输出是什么？
3. 输出是给人看，还是给系统消费？
4. 是否需要外部知识？
5. 是否需要调用工具？
6. 是否有高风险动作？
7. 是否需要稳定格式？
8. 可接受延迟是多少？
9. 成本上限是多少？
10. 如何判断回答是好的？

如果这 10 个问题答不清楚，先不要选模型和框架。
